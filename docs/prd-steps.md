# PRD Creation Workflow — Steps Reference

Created during `bmad-create-prd` session on 2026-04-03.

---

| # | Step | What Happens |
|---|------|-------------|
| 1 | **Init** | Workspace setup — create output file from template, discover and load any existing input documents (briefs, research, project docs) |
| 2 | **Discovery** | Classify the project — type (web app, SaaS, API, etc.), domain (general, fintech, etc.), complexity, greenfield vs brownfield |
| 2b | **Vision** | Define the core problem being solved, value proposition, and target users |
| 2c | **Executive Summary** | Write the PRD opening summary that captures the essence of the product |
| 3 | **Success Metrics** | Define measurable goals — how you'll know the product is working (KPIs, adoption targets, etc.) |
| 4 | **User Journeys** | Map key flows a user takes through the product (e.g., create project → add task → assign → complete) |
| 5 | **Domain** | Surface domain-specific requirements, concerns, and knowledge relevant to the product area |
| 6 | **Innovation** | Identify unique angles, differentiators, and opportunities that set the product apart |
| 7 | **Project Type** | Capture type-specific requirements based on classification (e.g., browser support, responsiveness, accessibility for web apps) |
| 8 | **Scoping** | Define v1 boundaries — what's in, what's explicitly out, and how features are prioritized |
| 9 | **Functional** | Full breakdown of feature requirements — the core of the PRD |
| 10 | **Non-Functional** | Performance, security, scalability, reliability, and other quality targets |
| 11 | **Polish** | Review the full PRD, fill gaps, refine language, resolve inconsistencies |
| 12 | **Complete** | Final PRD output packaged and ready for handoff to architecture and UX phases |
